1. Title and introduction

Margin Maestro is a Restaurant Margin Intelligence platform for independent restaurant operators. It connects ingredient cost changes to dish profitability, recommended actions, and margin-aware pricing decisions — and answers a single operator question:

"What should I change right now to improve my margins?"

Who this guide is for

You are an owner/admin beta tester with full access to one restaurant. This guide is written so you can use the platform end-to-end without a separate walkthrough: setup, daily workflows, edge cases, and QA verification.

What Margin Maestro is — and is not

It isIt is not
A decision layer over your menu, recipes and ingredient costsNot a POS
A margin intelligence and recipe costing toolNot accounting software
An impact analyzer when supplier prices moveNot inventory management
A recommended-actions engine for pricingNot a supplier marketplace
A menu pricing audit trailNot an external menu publisher / no POS push

Beta status

Important

This is a working beta. Real beta data and test/demo data coexist on a single shared Supabase backend (intentional cost decision for the beta phase). Please treat the data as durable but expect that other beta accounts share the same database. Backups and append-only audit logs are in place.

Current accepted build: 3.4A — Atomic RPC Accepted. Apply Price and Recipe CSV Import's update path now write the dish menu price and the audit row in a single atomic SQL function call.

Next planned build: 3.1 — Transactional Invite Emails (deferred until acceptance pass).

2. Quick start checklist

Use this on your very first beta session. It walks you across the whole product in about 30–45 minutes. Tick items as you go.

  • Log in at https://margin-maestro.vercel.app with email/password or Google.
  • Create your restaurant (or select your active restaurant if you already have one).
  • Open Settings → General: restaurant name, currency, locale, target GPM, tax mode.
  • Review Settings → Units & Conversions: 8 units (Ct, Gr, Kg, Lb, Oz, Ml, Lt, Gl) present and read-only.
  • Review Settings → Menu Categories: default categories seeded; add/rename/deactivate as needed.
  • Review Settings → Suppliers: default suppliers seeded; add your real ones.
  • Create or import a handful of ingredients (Primary, Fixed, Intermediate).
  • Create or import a few recipes (1–2 intermediate, several dishes).
  • Go to Price Log and Initialize Baseline.
  • Run a Price Update Batch on one ingredient to test the cost-change flow.
  • Open Menu Analytics, verify COGS / GP / GPM / target status / suggested price.
  • Open Dish Analysis for a below-target dish; review breakdown and Margin Manager.
  • Click Apply Price on a below-target dish; confirm the success toast.
  • Open Dish Analysis → Menu price audit history panel; verify the new entry.
  • Generate an Impact Cascade from the latest price-update batch.
  • Generate Alerts from the Alerts page.
  • Review the Dashboard: alerts, KPIs, recommended actions.
  • Export CSVs from Settings → Import / Export.
  • Open the QA pages linked at the bottom of Settings → Developer QA, especially /qa-beta-launch and /qa-mvp-readiness.

6. Ingredients

Your ingredients are the master cost database that everything else costs against. The Excel-derived logic is preserved.

Ingredient types

TypeMeaningCost source
PrimaryPurchased from a supplier (e.g., olive oil, flour).Calculated from purchase price + quantity + adjustment.
FixedItems you simply enter a fixed cost-per-unit for (e.g., spice blend, condiment).Manually entered cost per recipe unit.
IntermediateProduced in-house by an intermediate recipe (e.g., sauce, dough).Computed from a linked Intermediate Recipe's cost per serving.

Key fields

  • Total cost — what you paid for the purchased quantity (Primary only).
  • Original quantity + Original UoM — how much you bought, in the supplier's unit.
  • Recipe UoM — the unit your recipes use this ingredient in.
  • Adjustment / waste — yield adjustment, typically between −0.99 and a positive number. Cannot equal −1 (would divide by zero).
  • Density (g/ml) — required for any mass ↔ volume conversion (e.g., grams of olive oil from a milliliter recipe quantity).
  • Recipe unit cost — final cost per recipe UoM, derived from the above.
  • Calculation status — valid / warning / error / pending.

Validation rules

  • Name required, unique per restaurant.
  • Primary: total_cost ≥ 0, original_quantity > 0, UoM codes valid.
  • Fixed: manual_recipe_unit_cost required.
  • Adjustment cannot equal −1.
  • Ct can only convert to Ct.
  • Mass ↔ volume requires density.
Ingredients testing checklist
  • Create a primary ingredient (e.g., Olive Oil, 5 Lt, $50, recipe UoM Ml, density 0.91). Confirm recipe_unit_cost is calculated.
  • Create a fixed ingredient (e.g., House spice blend, Ct, $0.10/unit).
  • Create an intermediate ingredient stub (status should be pending until a recipe links to it).
  • Edit a primary ingredient: change adjustment, confirm cost updates.
  • Try an invalid conversion (e.g., Ct → Kg). Should show a clear error.
  • Deactivate an ingredient — confirm it disappears from active lists but historical references remain.
  • Run /qa-ingredients and review PASS/WARN/FAIL.

7. Recipes

Recipes come in two kinds:

  • Dish recipes — what you sell. Have a menu price and feed Menu Analytics.
  • Intermediate recipes — internal preparations that produce an Intermediate ingredient (e.g., tomato sauce, pizza dough). The recipe's cost per serving becomes the linked Intermediate ingredient's recipe unit cost.

Recipe header

  • Name — unique per restaurant.
  • Kind — dish or intermediate.
  • Menu category — optional, picked from your category list.
  • Serving quantity + Serving UoM — defines what one "serving" is for cost-per-serving math. Must be > 0.
  • Menu price — dish only, ≥ 0. Apply Price and manual edits both write to an audit log.
  • Linked intermediate ingredient — intermediate only; tells the system which ingredient receives the computed cost.
  • Notes — free form.

Recipe lines

  • Each line = ingredient + quantity + UoM + (sort order).
  • Line cost = converted_quantity × ingredient.recipe_unit_cost.
  • Recipe COGS = sum of line costs.
  • Cost per serving = COGS / serving_quantity.
  • For dishes: GP = menu_price − cost_per_serving; GPM = GP / menu_price.

Intermediates flow into dishes

When you save an intermediate recipe, the linked Intermediate ingredient's cost state is recalculated (cost_source = calculated, status = valid if everything is fine). Any dish recipe that uses that intermediate ingredient will pick up the new cost on its next read.

Cycle detection

You cannot create a circular intermediate dependency (Recipe A's intermediate ingredient is consumed by Recipe B whose intermediate ingredient is consumed by Recipe A). The system blocks the save with a friendly error. Self-reference is also blocked.

Edit behavior

  • Editing a dish recipe's menu_price writes a menu_price_audit_log row with source manual_recipe_edit (best-effort; the audit row is not yet atomic with the rest of the recipe save — see Known limitations).
  • Editing a recipe's lines triggers cost recalculation on save.
  • Recipes are soft-deletable (no hard delete from the UI).
Recipes testing checklist
  • Create an intermediate recipe (e.g., Tomato Sauce). Add 3 lines. Link it to an intermediate ingredient. Confirm the ingredient cost updates.
  • Create a dish recipe that consumes that intermediate ingredient. Confirm COGS reflects both direct and indirect costs.
  • Edit a recipe line quantity; save; confirm new COGS.
  • Try to create a cycle — should fail with a clear error.
  • Edit a dish recipe's menu price; confirm a new entry shows up in the Dish Analysis audit panel.
  • Run /qa-recipes.

8. CSV Import / Export

Located under Settings → Import / Export. Owner/manager only.

8.1 Ingredient CSV Import Implemented

  • Required columns: name, type.
  • Optional: supplier_name, total_cost, original_quantity, original_uom_code, recipe_uom_code, adjustment, density_g_per_ml, manual_recipe_unit_cost, notes.
  • Validation: name + type required; Primary requires cost + quantity + UoMs; Fixed requires manual cost; adjustment ≠ −1; invalid UoMs rejected.
  • Duplicate handling: skip (default) or update existing.

8.2 Recipe CSV Import (two files) Implemented Update path atomic for menu_price

Recipe import takes two CSVs:

  1. Recipes — headers: name, kind (+ optional category_name, serving_quantity, serving_uom_code, menu_price, target_gpm (informational only), linked_intermediate_ingredient_name, notes, is_active).
  2. Recipe lines — required: recipe_name, ingredient_name, quantity, uom_code (+ optional notes, line_order).

Duplicate handling modes

  • Skip existing (default) — skip recipes whose names already exist.
  • Update existing — patch the existing recipe. Kind transitions blocked.
  • Block duplicates — fail the preview if any name already exists.

Line handling modes

  • Append lines (default) — keep existing lines, append new ones.
  • Replace lines — hard-delete existing lines and replace with imported set (UI confirms before applying).

8.3 Templates & preview

  • Both ingredient and recipe imports offer downloadable templates.
  • Every import runs a preview first — no DB write happens until you press Apply.
  • Preview rows have a status: valid, warning, error, or skipped.
  • Apply is disabled if any row is in error.

8.4 Errors vs warnings

ClassExamplesEffect
ErrorEmpty name/kind, invalid UoM, line ingredient does not exist, line quantity ≤ 0Blocks the row from being applied; preview total includes errors and disables Apply.
WarningUnknown category (resolves to null), target_gpm column (informational), menu_price on an intermediate (ignored), unknown linked intermediate ingredient nameRow still applies (no DB write blocked), but operator should review.

8.5 Existing ingredient requirement

Important

Recipe import does not create ingredients. If your recipe lines reference an ingredient name that doesn't exist (or isn't active), the row errors out. Import (or create manually) all ingredients first, then run the recipe import.

8.6 Exports

DatasetFile
Ingredientsingredients.csv

8.7 Formula injection protection

Every exported cell that begins with =, +, -, @ (formula-risky) is prefixed with a single quote so spreadsheet apps treat it as text — defends against CSV injection.

8.8 Limitations

  • Imports are client-orchestrated — multiple Supabase calls in a row. They are not a single atomic transaction. The recipe-import update path's dish menu_price change uses an atomic RPC; everything else is best-effort with clear partial-success reporting.
  • No XLS/XLSX/XLSM import.
  • No POS export.
  • Recipe import does NOT: create ingredients, create suppliers, create categories, write to ingredient_price_log, create price_update_batches, write billing rows, or publish to a POS.
CSV testing checklist
  • Download the Ingredient template, fill 3 rows, upload, preview, apply.
  • Re-import same file with mode skip — rows skipped, no changes.
  • Re-import with mode update existing — fields updated.
  • Submit a malformed CSV (missing required column); preview shows errors; Apply is disabled.
  • Download Recipe + Recipe Lines templates, fill in 1–2 dishes using existing ingredients, apply with append.
  • Re-run recipe import with replace lines; confirm the confirmation prompt.
  • Try importing a recipe line whose ingredient doesn't exist — error.
  • Export every CSV; open in a spreadsheet; confirm formula-risky cells are escaped (leading apostrophe).
  • Run /qa-import-export and /qa-recipe-import.

10. Apply Price and Menu Price Audit Trail

Where Apply Price appears

  • Menu Analytics — Apply link next to suggested price on below-target rows.
  • Dish Analysis → Margin Manager — Apply this price under the primary target GPM suggestion.

What Apply Price does Atomic (Build 3.4A)

  1. Owner/manager triggers Apply with a positive, finite price.
  2. A confirmation dialog shows old price → new price.
  3. The browser calls a single Supabase SQL RPC: apply_dish_menu_price_with_audit.
  4. Inside one database transaction the function:
    • Locks the recipe row, validates kind = dish, is_active = true, role is owner/manager.
    • Updates recipes.menu_price.
    • Inserts an audit row into menu_price_audit_log with source = 'apply_price', old/new price, delta amount, delta percent, context (origin, target GPM, suggested price, reason), and changed_by = auth.uid().
  5. Either both writes succeed or both roll back — no partial success.
  6. UI shows: "Menu price updated to $X and audit entry recorded."

What Apply Price does NOT do Out of scope

  • Does not publish to a POS, online menu, ordering platform, or marketplace.
  • Does not update ingredients.
  • Does not write to ingredient_price_log.
  • Does not create price_update_batches.
  • Does not create billing records.
  • Does not apply across multiple dishes in one click (one dish at a time).

Audit sources

SourceTriggerAtomic with the price update?
apply_priceApply Price action (Menu Analytics or Dish Analysis)Yes (Build 3.4A)
manual_recipe_editSaving a dish recipe with a changed menu priceNo — best-effort (OI-30)
importRecipe CSV Import, update path on dishesYes for menu_price (Build 3.4A); non-price fields still client-orchestrated
system / otherReserved for future automated processesn/a

Audit history panel

Opening Dish Analysis → /dish-analysis/$id shows a read-only "Menu price audit history" panel with up to 25 most-recent entries: changed at, source, old price, new price, delta, delta %. The panel refreshes after Apply Price.

Append-only design

  • menu_price_audit_log has no UPDATE policy and no DELETE policy.
  • The DB enforces new_menu_price > 0 and recipe_kind_at_time = 'dish'.
  • Corrections come from new audit rows, never edits.
Apply Price testing checklist
  • Apply a price from Menu Analytics — verify success toast and audit row in Dish Analysis.
  • Apply a price from Dish Analysis → Margin Manager — verify identical behavior.
  • Manually edit a dish recipe's menu price; confirm a manual_recipe_edit audit row appears.
  • Run a Recipe CSV Import with mode update existing where a dish's menu price changes; confirm a import audit row appears.
  • Try Apply Price as a viewer — should fail with a friendly permission error.
  • Run /qa-apply-price, /qa-menu-price-audit, and /qa-atomic-rpc — all PASS.

11. Dish Analysis

The per-dish workspace. Open from Menu Analytics or by going directly to /dish-analysis/$id.

Sections

  • Dish header — name, category, serving info, current menu price.
  • Ingredient breakdown — line-by-line cost contribution sorted by share. See which 2–3 lines dominate the COGS.
  • Profitability summary — COGS, cost per serving, GP, GPM, target GPM, status.

12. Price Log and Snapshot

The Price Log is your append-only ingredient cost history. The Snapshot is the last-known baseline state.

12.1 Baseline initialization

First-time setup captures the current state of all active ingredients:

  1. System creates a price_update_batches row with source = baseline_initialization, status = applied.
  2. For each active ingredient, inserts an ingredient_price_log row with event_type = baseline (old = new, delta = 0).
  3. Inserts ingredient_snapshots rows that capture the current cost state.

Run this once per restaurant before doing your first price update batch.

12.2 Price Update Batch flow

  1. Create a new draft batch (price_update_batches.status = draft).
  2. Add rows: pick ingredients, enter new total cost or new recipe unit cost.
  3. Preview the batch: status moves to previewed.
  4. Apply: status moves to applied. Each row inserts an ingredient_price_log entry with event_type = change, snapshot of name + supplier at time of change, and old/new costs.
  5. Or cancel: status moves to cancelled; nothing is written to the log.

12.3 Append-only rules

  • ingredient_price_log has no UPDATE policy and no DELETE policy.
  • Corrections are new rows with event_type = 'correction'.
  • Editing an ingredient directly does not write to the price log. Use batches.

12.4 Non-destructive baseline reset

If you need to reset the baseline (e.g., switching cost model), the system bumps baseline_version and writes new rows. Old history is never deleted.

Price Log vs Menu Price Audit

SurfaceTracks

ingredient_snapshotsIngredient cost baseline state.
price_update_batchesGrouping of supplier-price updates.
menu_price_audit_logDish menu price changes (Apply Price, manual edit, import).
Price Log testing checklist
  • Initialize baseline; verify one row per active ingredient in the log with event_type = baseline.
  • Run a Price Update Batch on 2–3 ingredients; verify status transitions draft → previewed → applied.
  • Verify that an applied batch wrote one change row per ingredient.
  • Confirm there is no edit/delete UI on log rows.
  • Cancel a draft; verify nothing is written.
  • Run /qa-price-log-snapshot.

14. Impact Cascade

Answers: "We just changed these supplier prices — which dishes are now in trouble?"

How it works

  1. Run a Price Update Batch first.
  2. Open Impact Cascade and generate a new run against the latest batch.
  3. The engine:
    • Finds the changed ingredients from ingredient_price_log.
    • Builds old / new cost maps.
    • Finds dish recipes consuming changed ingredients directly (line uses the ingredient) or indirectly (line uses an intermediate ingredient whose recipe consumes the changed ingredient).
    • Recomputes each affected dish's COGS, GP, GPM at old and new costs.
    • Stores results in impact_cascade_runs + impact_cascade_items.

Direct vs indirect paths

  • Direct: Ingredient → Dish (e.g., Mozzarella → Margherita Pizza).
  • Indirect: Ingredient → Intermediate Recipe → Intermediate Ingredient → Dish (e.g., Tomatoes → Tomato Sauce intermediate → Pasta dish).

Only dish recipes appear as final impact items. Intermediate recipes are traversed but not stored as separate impact rows.

What each row shows

  • Dish name + path (direct/indirect).
  • Affected ingredients (names at time of change).
  • Old COGS → New COGS, COGS delta.
  • Old GPM → New GPM, GPM delta, target status.
  • Suggested price.

Why it matters

This is the core promise of Margin Maestro: a supplier price moves, you see exactly which dishes need pricing attention — including dishes that don't even reference the ingredient directly because they consume an intermediate that does.

Impact Cascade testing checklist
  • Have at least one dish that uses an intermediate ingredient.
  • Bump the supplier cost on a primary ingredient used by that intermediate.
  • Apply the batch.
  • Generate Impact Cascade; confirm both the directly-affected dish(es) and the indirectly-affected dish appear.
  • Verify newly-below-target count makes sense.
  • Run /qa-impact-cascade.

16. Dashboard

The Dashboard is your alert-first overview — the page you should open first thing each day.

Sections

  • Alert-first header — critical alerts banner.
  • KPI cards — average GPM, below-target count, open alerts count, missing-price count, recent price changes, impacted dishes.
  • Active alerts — latest open alerts with severity and quick actions.
  • Menu health — priced / below-target / incomplete counts, worst margins list.
  • Price activity — latest batch, recent changes, cost spikes.
  • Impact cascade — latest run summary.
  • Recommended actions — contextual next steps with links.

Daily review workflow

  1. Open Dashboard.
  2. Read critical alerts banner; resolve or act.
  3. Scan KPI cards for anomalies (e.g., average GPM dropping).
  4. For each recommended action, open the linked review page (Menu Analytics, Dish Analysis, Impact Cascade).
  5. Apply prices where appropriate.
  6. Generate Alerts again after substantive changes.

Warnings

  • Dashboard is per-serving only — there is no monthly impact view because sales volume is not yet tracked.
  • Dashboard does not mutate data. It links to review pages; you apply changes there.
  • Dashboard does not auto-generate alerts.
Dashboard testing checklist
  • Confirm KPIs render numbers consistent with Menu Analytics counts.
  • Confirm critical alerts show in the banner.
  • Confirm price activity card reflects your most recent batch.
  • Confirm Impact Cascade summary matches the latest run.
  • Verify no mock-looking data is visible (numbers should match your actual records).
  • Run /qa-dashboard.

17. Team Management

Members

Settings → Team shows the list of restaurant members with their roles. Owners can change roles. The sole-owner protection rule prevents removing the last owner — there must always be at least one owner per restaurant.

Pending invitations

Each invitation row shows email, role, created date, expiration (default 30 days), and status (pending). Actions:

  • Copy link — re-copies the accept-invite URL.
  • Resend email — calls the transactional email Edge Function. Deferred until 3.1A Today it returns "provider not configured" — share the link manually.
  • Cancel — invalidates the invitation.

Accept invite flow

  1. Invitee signs up or logs in with the invited email.
  2. Invitee opens https://margin-maestro.vercel.app/accept-invite?token=….
  3. RPC accept_restaurant_invitation validates the JWT email claim against the invitation email and inserts a restaurant_members row.
  4. Idempotent — re-running the link if already a member does no harm.

Manual link fallback Always available

Important

The clipboard copy is the source of truth for sharing. Even when transactional emails go live, the manual link fallback remains so that an email-delivery hiccup never blocks an invitation.

Future transactional email delivery Build 3.1 acceptance pending

Once Build 3.1A is accepted, the Resend-powered Edge Function will send an email to the invitee automatically when the invitation is created (and on Resend-button clicks). Toast outcomes:

  • "Invite link copied. Invitation email also sent."
  • "Invite link copied. Email delivery is not configured yet, so share the link manually."
  • "Invite link copied, but the invitation email could not be sent. Share the link manually."
Team testing checklist
  • Invite an email you control as manager; copy the link from the toast.
  • Sign in as that user using the invited email; visit the accept link; verify membership.
  • Confirm the manager can edit operational data but cannot send invitations.
  • Invite another email as viewer; verify they cannot edit anything.
  • Cancel an unaccepted invitation; the row should disappear from pending.
  • Attempt to remove the last owner; should be blocked with a clear error.
  • Run /qa-team-management and (when 3.1A is live) /qa-transactional-invites.

21. Suggested beta test scenarios

Each scenario walks one end-to-end thread. Run them in order on a fresh restaurant for the cleanest signal.

A. New restaurant setup from scratch Admin
Objective: Validate first-run flow and default seeds.
Steps:
  1. Sign up; create restaurant "Bistro Demo".
  2. Open Settings → General; set currency, locale, target GPM (e.g., 0.70).
  3. Confirm seeded categories and demo suppliers; rename one of each to your own.
  4. Add 3 primary, 1 fixed, 1 intermediate ingredient manually.
  5. Add 1 intermediate recipe linked to the intermediate ingredient.
  6. Add 2 dish recipes (one consuming the intermediate ingredient).
  7. Open Menu Analytics; confirm both dishes show COGS/GP/GPM.
Expected: Restaurant has working analytics with no mock data.
If it fails, report: the step number, the specific error toast, and a screenshot.
B. Ingredient import → recipe import → analytics
Objective: Validate CSV templates end-to-end.
Steps:
  1. Download the ingredient CSV template; fill with 10 rows of mixed types.
  2. Upload, preview, apply (skip-duplicates mode).
  3. Download the recipe + recipe-lines templates; fill 3 dish recipes referencing those ingredients.
  4. Upload both, preview, apply (skip-duplicates, append-lines).
  5. Open Menu Analytics; confirm the new dishes appear.
Expected: All rows applied; new dishes have correct COGS/GP/GPM.
If it fails, report: the preview row that errored or warned, plus the CSV content.
C. Supplier price increase → price log → impact cascade → alerts
Objective: Validate the core "supplier moves, dishes are flagged" promise.
Steps:
  1. Initialize baseline (Price Log).
  2. Create a draft batch; raise one primary ingredient's cost by 25%.
  3. Apply the batch.
  4. Generate Impact Cascade.
  5. Verify all dishes (direct and indirect) consuming that ingredient appear with old/new COGS/GPM.
  6. Generate Alerts; expect ingredient_cost_spike and possibly dish_newly_below_target.
Expected: Cascade includes indirect dishes (intermediate path). Alerts appear and link to review pages.
If it fails, report: which dishes were missed and the recipe lines that should have brought them in.
D. Below-target dish → Dish Analysis → Apply Price → audit trail
Objective: Validate atomic price update + audit.
Steps:
  1. From Menu Analytics, pick a below-target dish.
  2. Open its Dish Analysis page.
  3. Use Margin Manager Apply this price at the primary target GPM.
  4. Confirm success toast: "Menu price updated to $X and audit entry recorded."
  5. Scroll to Menu price audit history; verify the new entry (source apply_price, old/new prices, delta).
Expected: Single audit row with non-null delta; no warning toast.
If it fails, report: whether the price did or did not update, and any audit-not-recorded toast (should not happen post-3.4A).
E. Team invite + role restrictions
Objective: Validate roles and RLS through real flows.
Steps:
  1. Owner invites a second email as manager; shares link manually.
  2. Second user signs up with the invited email; accepts invite.
  3. Manager edits an ingredient (allowed) and tries to send a team invite (should be blocked).
  4. Owner invites a third email as viewer; viewer signs in.
  5. Viewer attempts to Apply Price (should be blocked client-side with a friendly message and server-side by RLS).
Expected: Each role can only do what the table in §4 allows.
If it fails, report: the action attempted, the role, and whether the failure was a friendly message or a raw error.
F. CSV export and data review
Objective: Verify exports respect RLS and sanitize formulas.
Steps:
  1. Export ingredients, recipes, recipe lines, menu analytics, price log, alerts.
  2. Open each in a spreadsheet.
  3. Confirm no cross-tenant rows.
  4. Confirm any cell starting with =, +, -, @ is prefixed with ' (single quote) in the raw CSV.
Expected: Files match what you see in-app; no formulas evaluated.
If it fails, report: the offending row and which export.
G. Negative testing
Objective: Verify the app fails safely.
Steps + Expected:
  • Upload a CSV with missing required column → preview errors, Apply disabled.
  • Recipe-import a line referencing a non-existent ingredient → row errors.
  • Set an ingredient adjustment of −1 → validation error.
  • Set a Ct ingredient with recipe UoM = Kg → conversion error.
  • As a viewer, hit owner-only endpoints (e.g., add supplier) → blocked.
  • Create a dish with no menu_price → Menu Analytics shows "Missing price".
  • Create a dish with one ingredient in cost-state error → "Incomplete costing".
Expected: All errors are friendly; no raw DB or stack traces.
If it fails, report: the raw error text and the action that triggered it.

  • Stripe verification deferred (OI-17, Build 2.2B). Deferred

  • Manual recipe edit audit is not atomic with the rest of the recipe-fields update (OI-30). If the audit insert fails the recipe save still committed. Limitation
  • Recipe import create path is not atomic across recipes, lines, and audit (OI-29). Update path's menu_price column is atomic. Limitation
  • Ingredient CSV import is not atomic — client-orchestrated across rows. Limitation
  • No XLS / XLSX / XLSM import — CSV only.
  • No POS or external menu publishing. Margin Maestro is a decision layer.

  • No multi-location advanced management. Restaurant switcher works; cross-location reporting does not.

  • No real payment testing unless Stripe test mode is explicitly configured for this project.
  • Custom unit management is not exposed — 8 canonical units only.
  • Google OAuth production hardening pending (OI-21). Works in beta.
  • Per-serving metrics only — no monthly impact view (no sales volume data tracked).

23. Glossary

COGS — Cost of Goods Sold
Sum of all ingredient line costs for a recipe at current ingredient costs.
GP — Gross Profit
For a dish: menu_price − cost_per_serving.
GPM — Gross Profit Margin
For a dish: GP / menu_price, expressed as a decimal (e.g., 0.72 = 72%).
Target GPM
Your restaurant's chosen target margin, set in Settings → General. Used everywhere "below target" is evaluated.
Recipe Unit Cost
The cost of one unit of an ingredient in the unit your recipes use it in (e.g., $0.005 per ml of olive oil). Derived for primary ingredients, entered for fixed ingredients, computed for intermediate ingredients.
Primary Ingredient
An ingredient you purchase from a supplier. Cost is derived from total cost + original quantity + adjustment + UoM conversion (and density if mass↔volume).
Fixed Ingredient
An ingredient with a manually entered cost per recipe unit. No supplier purchase math.
Intermediate Ingredient
An ingredient produced in-house by an Intermediate Recipe. Its recipe unit cost is the recipe's cost per serving.
Dish
A recipe of kind dish — what you sell. Has a menu_price and feeds Menu Analytics.
Baseline
The starting state for ingredient costs. Created by Baseline Initialization. Can be reset non-destructively (bumps a version number, never deletes history).
Snapshot
Immutable point-in-time capture of ingredient cost state, written at baseline initialization (and at non-destructive baseline reset).
Price Log
Append-only history of ingredient cost changes, in ingredient_price_log. No edits, no deletes. Corrections are new rows.
Impact Cascade
The traversal that takes a list of changed ingredients and surfaces every dish whose COGS, GP, GPM, or target status moves — including dishes that consume the ingredient indirectly via intermediate recipes.
Alert
An operator-facing signal generated from margin/cost issues (below-target, ingredient spike, missing price, etc.). Has a status workflow open → acknowledged → resolved (or dismissed).
Apply Price
The action that updates a dish's menu_price and inserts an audit row in one atomic SQL RPC call.
Menu Price Audit
Append-only history of dish menu price changes, in menu_price_audit_log. Sources: apply_price, manual_recipe_edit, import, system, other.
RLS — Row-Level Security
The Supabase / Postgres policy mechanism that enforces tenant scoping and role-based write rules at the database, not the client. Every tenant-owned table in Margin Maestro has RLS enabled.
Owner / Manager / Viewer
The three per-restaurant roles. See §4 for the access table.
